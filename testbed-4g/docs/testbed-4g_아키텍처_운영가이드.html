1바뀐 아키텍처 구조

기존 v0(브로드캐스트 MAVLink, 위성 RF)와 별개로, 셀룰러망을 실제 C2 매질로 태우는 4G 트랙이다. 핵심 설계 결정 3가지가 구조를 규정한다.

핵심 설계 결정

  • 멀티UE = multi-eNB CELL — srsRAN ZMQ는 1 eNB = 1 UE. GNU Radio 브로커(최대 난점) 대신 UE 수만큼 eNB를 띄워 각 쌍을 검증된 단일 구성으로 만든다. 모두 같은 EPC(MME)에 S1AP로 붙고, 같은 APN 풀 → UE-to-UE 도달(C2/공격 전제).
  • GPS off-셀룰러 GPS — C2만 셀룰러(serial0=tun_srsue), GPS는 비셀룰러 dahnet(serial2). gps_inject.py가 SIMSTATE+VFR_HUD를 GPS_INPUT으로 @10Hz 주입. 채널을 분리해 "C2망은 정상인데 GPS만 스푸핑" 같은 공격을 표현.
  • EPC 슬림 CELL — 4g-volte-deploy.yaml의 코어 9개만 --no-deps로 기동(IMS/osmo 제외). MME의 sgsap 블록은 자동 제거(S1 Setup 지연 방지).

주요 통신 흐름

  • Attach(부착): UE →(ZMQ RF)→ eNB →(S1AP 36412)→ MME →(S6a 3868)→ HSS 인증 → 베어러(GTP-U 2152) → tun_srsue IP 할당 = G0
  • C2: GCS(gcs-ue) ↔ SITL(uav-ue) 14550 — 셀룰러(tun_srsue) 위 양방향 = G1
  • GPS: gps_inject(dahnet) → SITL 14560 — 비셀룰러, arm→takeoff→land 폐루프 = G2
  • ARIA: 평문 14550 → 프록시 암호화 → 14555(셀룰러) → 복호 → 14550 — 서명(무결성)+ARIA(기밀성) 상보

🔎 운영 팁 — 트래픽 관측: docker exec srsue_zmq tcpdump -nA -i any udp port 14550(평문 MAVLink) vs ... udp port 14555(ARIA 적용 시 암호문). 셀룰러 링크에 평문이 안 보이면 암호화 정상.

⚠️ 공격 표면(TM1 동거 UE 주입=UE-to-UE 14550 · TM2(a) PFCP 8805 · TM3 S1-U 2152 변조)은 참조용 매핑일 뿐, 실제 공격 실행은 사용자 몫(역할경계).

4AWS 공유 EC2 연동

왜 AWS인가. srsRAN+Open5GS는 SCTP 커널모듈 + TUN + Docker host networking을 요구한다. 애플 실리콘 Mac은 Docker가 경량 VM 안에서 돌아 이 셋이 막혀 사실상 구동 불가. EC2 Ubuntu는 modprobe sctp가 되고 host-net이 네이티브 → 로컬 WSL2와 동일 구성을 그대로 재현. RF도 SDR 하드웨어 없이 ZMQ 소프트웨어 RF라 클라우드에 그대로 올라간다.

🍎 Mac (특히 애플 실리콘)

  • SCTP 커널모듈 — Docker VM 커널에 없음 (MME S1AP 불가)
  • /dev/net/tun — 경량 VM 격리로 UE TUN 생성 막힘
  • host networking — macOS Docker는 host-net 미지원
  • arm64 → amd64 이미지 에뮬레이션 느림/불안정

☁️ EC2 Ubuntu (x86)

  • SCTP — modprobe sctp 네이티브, 재부팅 자동로드
  • /dev/net/tun — 커널 디바이스 정상
  • host networking — Linux 네이티브
  • 로컬 WSL2와 동일 구성을 그대로 재현 · ZMQ RF라 SDR 불요

⚠ terminate ≠ stop — terminate는 EBS까지 삭제해 이미지·DB가 소멸한다. 반드시 stop만 사용.

stop → start → 재기동

# 안 쓸 때
aws ec2 stop-instances  --instance-ids <ID>
# 다시 쓸 때
aws ec2 start-instances --instance-ids <ID>
# SSH 후 원스텝 재기동
bash scripts/aws-resume.sh

이미지·DB는 EBS 보존 → rebuild 불요. srsRAN attach만 재수립. terminate 금지(디스크 소멸), 항상 stop.

유휴 자동 종료 (비용)

# 5분 주기 cron (root)
sudo crontab -e
*/5 * * * * .../scripts/\
  aws-idle-autostop.sh \
  >> /var/log/dah-autostop.log 2>&1

로그인 0 + 15분 loadavg<0.4 연속 3회 → 자동 stop. 켜두고 깜빡해도 요금 폭주 방지.

신규 계정 주의 — vCPU 할당량. 새 계정은 "Running On-Demand Standard" 한도가 기본 5 vCPU라 c6i.2xlarge(8)·4xlarge(16)가 시작 실패할 수 있다. Service Quotas ▸ EC2에서 16(또는 32)으로 증가 요청. + 종료 동작 = Stop, 종료 방지 = 활성화 권장.

docker rm -f dah4g-sitl                      # SITL만 제거(공격 대상 리셋)

docker stop $(docker ps -q)                  # 전체 정지(인스턴스 stop 전 불요 — stop이 알아서)
개념 정리. 이 테스트베드에서 docker build는 ①스크립트 내부(open5gs·srsRAN, 자동)와 ②air 이미지(수동 1회) 두 군데서만 일어난다. 나머지는 전부 docker compose ... up -d(빌드 아님, 이미 빌드된 이미지 기동)와 docker run(SITL/GCS/GPS)이다. 그래서 일상 운용은 build가 아니라 up/run/logs가 대부분.
