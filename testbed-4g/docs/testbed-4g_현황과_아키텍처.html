1. 한눈에 보는 현황

설계·문서·v0 산출물 골격은 완성되었으나, 실제 기동·검증은 0%다. 남은 일은 "코드를 더 짜는 것"이 아니라 EC2에서 실측값으로 채우고 게이트를 통과시키는 것이다. 로컬은 셸/Docker가 차단되어 testbed-4g/는 EC2(Ubuntu) 튜닝을 전제한 v0 골격이다.

2. 진행 게이트 (G0 → G2)

설계/골격 ✔G0 ✔G1 ✔G2 폐루프 ✔

G0·G1·G2 전부 통과(2026-07-01, 로컬 실측) — 테스트베드 핵심 완성. G0: EPC(슬림9)+eNB+UE attach. 멀티UE=multi-eNB(브로커 회피)+UE-to-UE(UDP14550). G1: ArduPilot SITL↔GCS 양방향 MAVLink C2(HEARTBEAT/COMMAND_ACK) over LTE. G2: GPS=dahnet(비셀룰러)+C2=셀룰러 동시로 arm→takeoff(9m)→land 폐루프 완주. 검증 중 실버그 3개(CRLF·SGsAP·GPS freshness245ms→10Hz) 발견·수정.

게이트판정 기준상태난이도
G0 — LTE 기질 srsue RRC Connected + attach successful + tun_srsue IP → UAV-UE→GCS-UE ping(셀룰러 경유) 미통과 하 (단일UE)
G1 — C2-over-LTE SITL의 UDP C2가 tun_srsue로 GCS-UE IP 왕복 (gcs_cli ACK) — bridge 적용 미통과 중
G2 — 폐루프 + GPS off-셀룰러 GPS_INPUT=dahnet, C2=LTE 동시 정상 + mode→arm→takeoff→land + EKF 추종 미통과 중상 (멀티UE 전제)

3. 구상한 최종 Testbed 아키텍처

⚠️ 아래 다이어그램은 구현 전 설계안(1 eNB + gr-broker)입니다. 실제 구현은 multi-eNB(브로커 회피)로 바뀌었습니다 — 최종 아키텍처·컨테이너 역할·통신구조·사용법은 testbed-4g_아키텍처_운영가이드.html(multi-eNB 최종형)을 보세요.

4개 계층(애플리케이션 → UE → RAN → EPC 코어)으로 신호가 내려가는 구조. GPS는 셀룰러에 태우지 않고 별도 채널(dahnet)로 SITL에 직접 입력한다. 우측에 공격 표면(TM1/TM2/TM3)이 각 계층에 매핑된다.

최종 아키텍처 — 신호 흐름(↓) · GPS 비셀룰러 채널(좌) · 공격 표면 TM1/TM2/TM3(우, 출처 = Sonaglio et al. 논문 §아래)
핵심 설계 원칙 3가지 — ① GPS는 셀룰러 밖(dahnet으로 SITL 직접 입력) → C2만 LTE에 태운다. ② Open5GS EPC = CUPS 구조라 Sxa/Sxb에 PFCP가 존재 → TM2 가용성 공격이 4G에서도 성립(이전 "5G 전용" 라벨 정정). ③ srsenb가 PDCP 종단 → 이후 S1-U GTP-U는 평문 → TM3가 Spiderweb(실 4G 작전)에 가장 직결.

3-B. 논문 기반 & 위협모델 매핑

이 테스트베드의 위협모델(TM1/TM2/TM3)은 아래 두 논문에서 그대로 가져왔다. 단, 논문 테스트베드는 5G SA(Open5GS 5GC + UERANSIM)이고 본 구현은 4G/LTE(Open5GS EPC + srsRAN)로 의도적 적응했다 (근거 = Operation Spiderweb의 실 4G). 그래서 TM별 충실도가 다르다.

논문 A
Impact of 5G SA Logical Vulnerabilities on UAV Communications: Threat Models and Testbed Evaluation
W. C. Sonaglio, Á. L. R. Ferraz, L. A. Pereira Jr. — Aeronautics Institute of Technology (ITA), Brazil
→ TM1/TM2/TM3 위협모델의 원출처. 테스트베드 = Open5GS + UERANSIM + Kubernetes (UAV–GCS over 5G SA).
논문 B
When Connectivity Is Not Enough: Cross-Layer Attacks on UAV C2 over 5G
W. C. Sonaglio, Á. L. R. Ferraz, A. E. Melo, M. E. de Souza (ITA), G. Noubir (Northeastern Univ.), L. A. Pereira Jr. (ITA)
→ 확장판. "연결성(등록·PDU세션·IP도달) ≠ 안전한 폐루프 제어". MAVLink over 5G User Plane. 5개 견고성 이슈 공개(3 CVE 발급, 2 신청). arXiv:2603.04662 (v1=A, v4=B).
위협모델논문 정의 (무엇을 stress)논문 현상testbed-4g 대응4G 충실도
TM1
동거 UE		 User Plane 공유 큐잉 — co-tenant UE가 같은 UPF 경합 stale 텔레메트리 · heavy-tail 지연 같은 APN/PGW UE-to-UE → SysID 255 주입(CWE-306) 충실
단 멀티UE 필요
TM2
코어 접근		 SBI(서비스기반 인터페이스) — 이동 중 Control Plane 불안정 (5G 전용) 핸드오버 후 failsafe SBI 부재 → 4G는 PFCP 세션삭제(EPC=CUPS) 또는 MME/핸드오버로 대체 적응
SBI 경로 🔴불가
TM3
탈취 RAN		 RAN — gNodeB PDCP 복호 후 명령 변조 (UP 무결성은 3GPP 선택사항→흔히 off) 내비게이션 하이재킹 eNB PDCP 종단 → S1-U GTP-U 평문 → NFQUEUE 변조 최충실
Spiderweb 직결
왜 멀티UE가 최대 난점인가 — 논문과의 연결. 논문은 UERANSIM을 써서 UE를 독립 프로세스로 무한정 띄웠다(co-tenant TM1이 거저). 본 구현은 srsRAN ZMQ로 피벗하며 "1 eNB = 1 UE" 제약이 생겼고, 그래서 멀티UE가 어려워졌다. TM1은 co-tenant(≥2 UE)가 본질이므로 멀티UE 없이는 재현 불가 → 해법으로 "기지국을 UE 수만큼"(multi-eNB) 토폴로지를 택해 브로커 없이 각 쌍을 검증된 단일 UE 구성으로 만든다. (논문 정합이 더 중요하면 5G/UERANSIM 트랙 testbed-5g/가 1:1.)

5. 구현해야 할 부분 (잔여 작업)

코드 골격은 있으나 그대로는 안 돈다. 아래는 EC2에서 채워야 동작하는 구체 항목들이다.

A. 환경 부팅 — 게이트 0 (인프라)

  • EC2 c5.2xlarge / Ubuntu 22.04 — modprobe sctp, /dev/net/tun, docker, EBS 40GB+, 인바운드 SSH만.
  • docker_open5gs clone → EPC 기동 → 실측값 확보: EPC_NET · MME_IP · SRSRAN_IMAGE 태그 → .env.4g 작성. 현재 .env.4g.example 값은 전부 추정치 (10.10.0.5 / srsran:latest 등)
  • scripts/provision.sh로 HSS에 IMSI 3개 등록 (key/opc는 user_db.csv와 일치).

B. 골격 안의 명시적 미완성 (플레이스홀더 5곳)

① ★ gr-broker 진입점 — 더미 명령

docker-compose.4g.yml:78 — ["python3","/opt/srsran/zmq_broker.py","--help"]

멀티UE GNU Radio 브로커의 실제 진입점·포트·샘플레이트 정합으로 교체. docker_open5gs의 멀티UE 예제 차용. 전체 일정 최대 난점 — 실패 시 2 UE(UAV+GCS)로 범위 축소.

② SITL 브리지 명령 — 의사코드

bridge.4g.yml:19-22 — "ardupilot SITL 기동 시 …" (실 명령 아님)

기존 testbed start-sitl.sh의 실제 인자에 --serial1=udpclient:<GCS_UE_IP>:14550 (B3) 반영해 배선.

③ enb.conf — sib/rr/rb.conf 의존

srsran/enb.conf:16-19 — 이미지 기본 예제 의존(파일 없음)

EC2 이미지 내 /etc/srsran/에 해당 파일 존재·정합 확인. ZMQ rx_port 매핑이 단일UE 직결 전제라 멀티UE 시 broker로 재배선.

④ provision.sh — DBCTL 경로 환경의존

scripts/provision.sh:9 — DBCTL=open5gs-dbctl 기본값

실제 컨테이너명으로 확정(예 docker exec docker_open5gs-dbctl-1 open5gs-dbctl). 버전별 add vs add_ue_with_apn 분기.

⑤ .env.4g 후속값 — G0 통과 후 채움

.env.4g.example:14-18 — UAV_UE_IP / GCS_UE_IP / AIR_IMAGE / TOOLS_IMAGE

attach 성공 후 실제 tun_srsue IP 확인해 bridge용으로 채움.

C. 게이트 통과 (검증)

  • G0 — 단일 UE(3-A, broker 불필요)로 LTE attach + UE↔UE ping. 쉬움
  • 3-B 멀티UE — gr-broker로 3 UE 합성(TM1 동거 성립). 최대 리스크
  • G1 → G2 — C2-over-LTE → 폐루프(arm→takeoff→land) + GPS off-셀룰러 동시 정상.

6. 공격 표면 — 4G 수행가능성

아래 TM1/TM2/TM3는 §3-B의 Sonaglio et al. 논문 위협모델을 4G로 적응한 것이다(TM2는 SBI→PFCP 대체).

7. 정직한 한계

  • 전이간극: ZMQ는 RF 시뮬(deterministic), RTT는 SW 경로 → 아날로그 검증 ≠ 실물 취약 증명.
  • 멀티UE ZMQ broker가 단일 최대 리스크. 단일 UE는 쉬우나 3 UE는 GNU Radio 정합 튜닝 필요.
  • srsRAN 4G는 유지보수 모드(srsRAN Project는 5G) — LTE 테스트엔 동작하나 버전 확인.
  • config/포트/IP·이미지 태그는 EC2에서 확정(현재 플레이스홀더).
  • 5G 전용 SBI NF-크래시는 본 테스트베드 범위 외(EPC에 SBA 없음).
  • 역할경계: 인프라/배선만 구축 — 실제 공격 실행은 사용자.
