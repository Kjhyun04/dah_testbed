팀원 접속 가이드

공유 EC2(테스트베드 서버)에 팀원을 초대하고 접속하는 방법. 팀원용과 관리자용 트랙으로 나뉩니다.

📌 서버 주소(고정 IP): <여기에 EIP 기입 — 예: 54.88.236.111> · 로그인 사용자: ubuntu
※ 반드시 탄력적 IP(EIP) 주소를 공유할 것 — 자동할당 IP는 재시작 시 바뀜.
🔑 키 3종 이해 🟢 팀원용 🔵 관리자용 🖥 웹 UI 터널 🤝 협업 규칙 🛠 트러블슈팅

🔑 먼저: 키 3종 구분

헷갈리기 쉬운 세 가지를 구분하면 나머지가 쉽습니다.

키누구 것어디서 만드나인스턴스 새로 만들면?
id_ed25519 (개인키/공개키)팀원 개인 팀원 본인 PC에서 ssh-keygen 1회❌ 안 바뀜 — 평생 재사용
*.pem (AWS 키페어)서버 소유자(관리자) AWS 콘솔에서 인스턴스 생성 시⭕ 새 인스턴스마다 다를 수 있음
호스트 키서버 자신인스턴스가 자동 생성새 인스턴스면 바뀜(known_hosts)
핵심. 팀원은 id_ed25519를 한 번만 만들고 공개키(.pub)만 관리자에게 전달한다. 서버를 새로 만들어도 팀원 키는 그대로 — 관리자가 새 서버에 공개키를 다시 등록만 하면 된다. 관리자의 .pem은 팀원에게 나눠주지 않는다.

팀원용 접속 준비 & 접속

1내 SSH 키 생성 (본인 PC에서, 최초 1회) — Windows/Mac/Linux 공통

ssh-keygen -t ed25519 -C "이름@dah_testbed"
# 프롬프트는 전부 Enter(기본 경로/무암호). 이미 있으면 새로 안 만들어도 됨.

생성 위치: ~/.ssh/id_ed25519(개인키·비공개) + ~/.ssh/id_ed25519.pub(공개키·공유가능)

2공개키를 관리자에게 전달 (이 한 줄만 복사)

# Mac/Linux
cat ~/.ssh/id_ed25519.pub
# Windows PowerShell
type $env:USERPROFILE\.ssh\id_ed25519.pub

출력된 ssh-ed25519 AAAA... 이름@dah_testbed 한 줄을 관리자에게 보냄. + 내 공인 IP도 함께 전달(브라우저에서 checkip.amazonaws.com 접속 → 나온 주소).

절대 금지: 확장자 없는 id_ed25519(개인키)를 공유하지 말 것. .pub만 전달.

3접속 (관리자가 "등록 완료" 하면)

ssh -i ~/.ssh/id_ed25519 ubuntu@<서버 EIP>
# 처음이면 "Are you sure...?" → yes

Windows에서 Permission denied·bad permissions가 뜨면 아래 트러블슈팅 참고.

관리자용 팀원 추가 (2가지만)

팀원에게 받은 ①공개키 ②공인 IP 로 아래 두 가지를 하면 끝.

A공개키 등록 (서버 EC2 안에서)

# 받은 공개키 한 줄을 추가 (여러 명이면 반복)
echo "ssh-ed25519 AAAA...받은공개키... 이름@dah_testbed" >> ~/.ssh/authorized_keys

# 권한 유지(중요 — 홈/.ssh 권한이 열리면 sshd가 접속 거부)
chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys

B보안그룹에 팀원 IP 허용 (AWS 콘솔)

EC2 → 보안 그룹 → 이 인스턴스의 SG → 인바운드 규칙 편집 → 규칙 추가:

  • 유형 SSH · 포트 22 · 소스 <팀원IP>/32 · 설명 "이름"
집/카페 IP는 바뀔 수 있어 바뀌면 이 규칙을 갱신해야 한다. 키 인증이 필수라 0.0.0.0/0도 치명적이진 않으나 인터넷 스캔에 노출되므로 IP 지정 권장.
팁. 새 서버로 이전할 때도 팀원은 아무것도 안 해도 된다 — 관리자가 새 서버에서 A(공개키 목록 붙여넣기) + B(IP 규칙)만 다시 하면 된다.

🖥 웹 UI 접근 (SSH 터널)

noVNC·로그뷰어·Open5GS WebUI는 서버 루프백에만 열려 있어(보안), SSH 터널로만 본다. 접속 권한만 있으면 추가 설정 없음.

ssh -i ~/.ssh/id_ed25519 \
  -L 6080:localhost:6080 \   # Gazebo noVNC  → http://localhost:6080/vnc.html
  -L 8080:localhost:8080 \   # 로그뷰어      → http://localhost:8080
  -L 9999:localhost:9999 \   # Open5GS WebUI → http://localhost:9999 (admin/1423)
  ubuntu@<서버 EIP>

터널이 열려 있는 동안 로컬 브라우저에서 localhost:포트로 접속. 세션 닫으면 터널도 닫힘.

🤝 협업 규칙

모두 같은 EPC·컨테이너를 공유하므로 충돌을 피하는 규칙이 필요하다.

  • 동시 1실험 + 나머지는 관찰 — 한 명이 기동, 나머지는 같은 화면을 본다.
  • tmux 공유 세션: 
    기동자:  tmux new -s dah
관찰자:  tmux attach -t dah        # 보기만: tmux attach -r -t dah
분리:    Ctrl+b 뗀 뒤 d
  • 서로 다른 실험을 동시에 돌리면 MME/포트가 겹치니 피할 것.

🛠 트러블슈팅

증상원인 / 해결
Windows: bad permissions / UNPROTECTED KEY 키 파일 권한 과다. PowerShell:
icacls "키경로.pem" /inheritance:r
icacls "키경로.pem" /grant:r "$($env:USERNAME):(R)"
Permission denied (publickey) ① 관리자가 내 공개키를 아직 등록 안 함(A) ② 내 IP가 SG에 없음(B) ③ 잘못된 개인키 지정(-i 경로 확인)
접속 자체가 안 됨(타임아웃) 서버가 중지(stop) 상태이거나, IP가 바뀜(EIP 미사용). 관리자에게 서버 상태·주소 확인
REMOTE HOST IDENTIFICATION HAS CHANGED 서버를 새로 만든 경우(호스트 키 변경). known_hosts에서 해당 줄 삭제 후 재접속
웹 UI 접속 안 됨포트는 루프백 전용 — 반드시 SSH 터널로 접근
